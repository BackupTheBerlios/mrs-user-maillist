Op 1-mrt-2007, om 13:46 heeft Guy Bottu het volgende geschreven:
>
> The current version of the MRS 3 Web interface does not allow to write
> queries with logicals, presumably because the query is not handled
> as it is to the search engine (using a Perl script it does work). I
> understand you prefer to develop the new interface rather then to
> debug the old one ; still, it is maybe just a few lines of code to  
> change.
> Then it would be possible to type queries like :
> os:carica AND os:papaya AND (de:protease OR de:proteinase)
> This the only thing we wait for before shifting to MRS 3 on the BEN  
> site.

OK, fair enough... I'll fix this.

The reason I didn't implemented this in the first place was that I  
thought it would be cool to be able to copy/paste a piece of text  
from a record and then be able to find this record back again. That  
means skipping over punctuation and such without producing errors.

But that doesn't work anyway when there's a colon or minus somewhere  
in the copied text. :-/

(Did you know you can search for -human to exclude all records  
containing the word human e.g.?)

So ok, I'll do this. The question though is, what should be regarded  
as boolean operators. The obvious candidates are AND, OR and NOT. In  
fact, AND is implicit, only OR is needed. NOT can be replaced by  
'-' (as it is now) or even '!'. And then AND and OR could have  
synonyms in the form of '&' and '|'.

hmmm

When we choose to accept AND, OR and NOT at least these will have to  
be case sensitive. Otherwise the grammar gets too ambiguous.

But is it an option to allow only '-' and '|' as operators (along  
with parenthesis of course). Your query would then be entered like:

"os:carica os:papaya (de:protease | de:proteinase)"

Is that acceptable?
