First, formatting. That's the process of taking a raw document (entry in a databank) and turn it into something more readable and useful. In the previous versions of MRS this was done by the parser scripts (written in Perl) and handled by the server. The new MRS will leave this task to the browser. Just look at the source code for a SwissProt entry in http://mrs.cmbi.ru.nl:18090/

That's cool. Browsers can do a lot more these days. Especially now that Microsoft seems –thankfully– keen to shut down IE6 for ever




The memory usage of MRS is something that is tied to the speed of indexing. It is something that cannot be changed unless you would accept a tenfold increase in indexing time (at least tenfold). I don' t know if the amount of RAM needed has changed in the new version, have to check that.

Hmmm…. Is there a way to redirect, during runtime, where the indexing may be done? For example I got half a terabyte of SAS 6GB/sec at 15krpm in raid6+0 max raid level. I also have a SATA III controller with 2gb cache with raid 6+0 max as well, and a bunch of intel SATA III SSDs . It's not as fast as DDR3-10600 but still would like to see the performance difference. For me, it makes more sense to buy fast storage than buy more memory for my (limited) resources, as our mrs installation in in VirtualBox and it has to play along with all the other VMs.


On the estimate of disk usage, a rough guess is that the generated MRS file is less than the size of the uncompressed raw data and sometimes even the half of that. The new MRS will give you a couple of progress bars that will help you estimate how long the indexing will take.

I have to apologize  for the silly question, but can you please remind me how does MRS schedule the indexing? Is it a cronjob, or is it done in the makefile, after updating is done?

I am asking because there is this technique VirtualBox called "memory ballooning". It is basically run-time dynamic memory allocation to a VM, much as if you were to stick another memory board on a mainframe and it would happily start using it, after initialization.

If there is a predictable way to tell/query the M6 server when it is thinking about starting indexing and when it is going to finish,  I could then use Memory ballooning in VirtualBox to allocate more memory by taking memory away from other virtual machines, or shutting them down for the duration of indexing.

Also, hmmm, I think I can write a nagios module to check the status of the indexing…




SOAP will stay. We use it extensively in house and e.g. the MRS::Client script of Martin Senger depends on it. For those who dislike SOAP, the same code supports REST style calls for most of the functions. I use JSON in the web interface. I could add a new API for use with JavaScript, but you can use SOAP from JavaScript as well so I don't see much added value here.
