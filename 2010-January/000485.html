Op 29-jan-2010, om 14:21 heeft Oscar Eriksson het volgende geschreven:
>
> I'm trying to set up a mrs-web site locally on a mac osx 10.6 server
> and I've come across some problems. I just want to declare that I am
> no biologist, so my questions may be stupid.

Neither am I :-) Didn't keep me from writing MRS.

> I managed to build and run the site if I force it to run in 32-bit
> architecture (intel x86) by using perl5.8.9, but not everything works
> as I suppose it should.

32 bit is not good enough. The biological databanks are so huge you  
can forget to index those on a 32 bit machine. The smaller ones are  
doable though.

> First off, I could only build about half of the databases in
> databank.info file in the update_scripts folder. Is that normal or
> should I be able to build all databases in that file?

Well, that depends. Considering the 32bit addressing, the big ones  
will definitively fail. Most smaller ones should be OK though. You  
might want to take a look at the per databank log files that are  
written in /data/status/

> Secondly, when I run the mrs-web frontend I can't get the fasta
> information from the databases. I only gets the output:
> "...
> .pm' No such file or directory at /u
> sr/local/share/mrs/parser_scripts/MRSScript.pm line 45.
> "
> Is this right?

No. First of all, the snippet of data comes from a databank that does  
not contain sequences as far as I can see. That means there won't be  
a fasta either. But then the line:

 > Error in formatting entry: open 'MRS::Script::enzyme=HASH 
(0x848c110)/ID  1.1.1.5

worries me. No idea how you managed to get that one.

> Thirdly, when I try to run Clustal, I get the java exception:
> "javax.servlet.ServletException: No clustal executable specified in
> mrs config file"
>
> I have installed clustalw2 and the executables are located in
> /usr/local/bin/clustalw2 as specified in /usr/local/etc/mrs-config.xml
> by the tag <clustalw-exe>

The version of MRS you use most likely has a bug, the config file has  
two locations where the <clustalw-exe> tag occurs. One is in the base  
section, next to e.g. <scriptdir>. The other is in the <server>  
section for the clustalw server, next to <address>. I don't know  
which one is used in your version, but most likely it is the other  
one if you know what I mean. :-)

> And my last problem. When I try to run a blast, I get the output that
> the database isn't loaded.
> "Failed to submit blast job Databank gpcrdb was not loaded"
> I can only choose between two databases even thou I've successfully
> installed other data sources (according to the logfiles).

Remember that you have to specify the databanks in each <server>  
section. The ones loaded by the <search> server are not automatically  
available to the <blast> server. Maybe this is why?
