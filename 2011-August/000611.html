Hei Maarten,

Thanks for a much better system (5). Throwing Java off the Server Side 
of things makes a big difference in response time for queries, as far as 
I can see with the dataset I have so far:

http://cn1.uio.no:18080/

I will blog the setup on a RHEL 6 platform (unless you have a proper 
4.4.5 gcc/g++ RHEL 5/CentOS 5 is out of the question), however on to a 
small issue:

In the server, I have setup the location of the BLAST related external 
binaries in the mrs-config file in the 'tools' section :

<!-- Tools section, specify external executables used by MRS -->
<tools>
<tool name="clustalw">/biotek/cn1/storage/mrs/external/clustalw</tool>
<tool name="blastall">/biotek/cn1/storage/mrs/external/blastall</tool>
<tool name="formatdb">/biotek/cn1/storage/mrs/external/formatdb</tool>
<tool name="fastacmd">/biotek/cn1/storage/mrs/external/fastacmd</tool>
</tools>

I know this works for basic blast queries because when I submit one, I 
can see the blastall executable referenced properly in the defined path 
('globin' as a search term, select a result from TrEMBL, click on Blast, 
verify the data entry is in Fasta format, submit BLAST query against 
'PDB'). My understanding is that BLAST-ing against PDB uses the external 
blastall binary, whereas other queries against say 'UniProt' appear to 
show the mrs-ws handling things.

Now, the error/question:
If I select a PDB entry and try to grab it by hitting BLAST, the system 
produces an error saying:
"Unfortunately, an error occurred. Maybe the following message may 
reveal what happened:

         The fastacmd executable '/usr/bin/fastacmd' does not seem to exist"

Obviously the system is referencing fastacmd on the wrong path and yes, 
I could resolve it with a symlink to the external binaries dir. My 
question is why, when you specify in the file the binaries, the BLAST 
button is still trying to see things under /usr/bin?
