Op 1-dec-2006, om 19:41 heeft Guy Bottu het volgende geschreven:

>>    1. the quest for zero-config (Maarten L. Hekkelman)
>> And so I can move to a zero-config situation where the files
>> available in the MRS_DATA_DIR define what you can see in the
>> interface. Only problem remaining is the extra info the config file
>> mrs.conf provided like e.g. the pretty name. I'm considering to store
>> this information using an XML formatted string in the databank. This
>> XML string will be generated by the parser and the interface can use
>> it to find out what the databank is all about.
>>
>> Is this an acceptable solution?
>
> I am not sure it is a good idea to drop the mrs.conf file. The
> maintenance of this file is anyway only a small effort. And it can  
> make
> sense to choose the order in which the databanks are displayed rather
> then having the list be generated automatically (alphabetically ?) ;
> it can also be useful to have some databanks not visible in the Web
> interface (e.g. because they are "under construction").

I was thinking about adding a group identifier to the databanks. That  
way you can group databanks together and the popup will only show  
'protein', 'nucleotide', 'gene', etc.

OK, I agree with you that it can be useful to have a config file to  
show only a subset, and use it to define the order. Maybe I can make  
it optional to have such a file.

> Taking about this : besides the pretty name of the databank, the  
> mrs.conf
> file also contains a hyperlink to the "production site", which is  
> also a
> useful information. It could be a bonus to have also a short  
> description
> text explaining what the databank is about.

This information should be stored inside the MRS file instead of a  
config file. Well, that's what I think now anyway. This way you can  
share databanks and the one receiving such a file has all the  
information available. I also consider storing the parser/formatter  
script (which is only one script now btw) inside the db file. For the  
same reason.
