Hi Maarten

see below for my comments; I've given them rates from 1 to 5, 5 being 
the highest priority
from my point of view..
> - call back for parser scripts
> 	The idea is to be able to do queries on other databanks from inside a 
> formatter script. This way you can e.g. enable or disable links to other 
> entries that may or may not exist.
>   
prio: 4
> - linking
> 	The SRS way of linking results from one databank to records in another.
>   
prio: 4
What I would like to have here, is not (like SRS) have just one way of 
linking 2 databanks, but several
ways of linking them... That also means expanding the query language to 
be able to express what  link route to
use... Also, one thing that would not be needed from my point of view, 
is the "shortest route" type of linking,
where you can link database A to Z while there are no direct links 
available and the system could get
their using a shortest route approach. Reason is, often you don't end up 
with the things you want; so you
just end up writing out the route in the query language from hop to hop...


> - more RDBMS like storage of records
> 	This may be necessary to do the linking right. It allows for more SQL 
> like queries and perhaps even nested queries. MRS already has so-called 
> meta data fields for storing extra information along with a document but 
> the implementation may need to be improved.
>   
Not sure if with RDBMS you mean a Read/Write system... that for me 
wouldn't be high prio (prio: 1).

prio: 5 Wrt the meta data fields, yes I would like to see that 
improved... so like the SRS way,
you could do getz -f 'id des aut' '[embl-des:prion]' e.g. and only those 
fields would be returned, similar to in
sql doing "select id, des, aut from embl where des like '%prion%';
Where the main important thing I think is, being able to basically - out 
of the MRS box - be able to just select
any field that the "parser" knows of... so e.g. also sequence field, 
without having to store it (from the admin/dev
point of view) in the meta data again... I guess just keeping pointers 
to those pieces of info in the file (cmp)
won't work since the data is compressed?

> - case sensitive indices
> 	Needed for the RDBMS feature and some databanks (wikipedia e.g.)
>   
in RDBMSystems it's configurable ; I believe in mysql the default is 
case-insensitive... So also for me, not
high prio. prio:1

> - DNA blast and all the translated versions.
>   
prio:2
Wrt amount of work, I think a "wrapper" approach would be a lot faster, also
for future developers, instead of re-implementing it. (I'm assuming 
currently
it's a kind of re-implementation...).

> - allow the storage of images inside a databank
> 	Should already be possible, but I never tried it... There's no way to 
> access such images from a web page yet.
>   
prio 1


> - more databanks
> 	
> I'm not sure if all of this is doable. And I'd like to hear where you 
> think priorities are. If you have additions to this wish-list, please 
> post them here.
>   
prio: 1 (but I'm biased ;) ).

thanks for asking Maarten,
regards,
Koen
