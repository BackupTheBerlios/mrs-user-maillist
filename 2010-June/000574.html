Many thanks for a quick reply.

I know that my examples with parenthesis seemed a bit artificial. The reason
I came across this issues is that I am updating the MRS::Client and trying
to add there a similar search string parser that is present in the mrs-web.
Therefore, I am trying to understand what is the "small language" behind it.

If I read it correctly, the result of parsing a search string (before
sending it to the MRS server) is a list of query terms (the ones that will
be searched for by the AND operator) and a filter string (that will be
applied additionally to the query terms). For example, I am getting these:

Search string                   Query terms                 Filter string
a AND b AND c                   [a, b, c]                   null
a AND NOT b                     [a]                         NOT b

So far, so good. But for a search string "a AND b OR c", I would expect that
"a AND b" should have precedence but it does not seem so:

Search string                   Query terms                 Filter string
a AND b OR c                    [a]                         b OR c

And now coming back to the parenthesis. If I would like to set the
preferences myself, I need the parenthesis: "(a AND b) OR c" (which does not
work; as reported). So that's why I came to the issue.
