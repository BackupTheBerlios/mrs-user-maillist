Folks, I would say that MRS needs a good base of 64 Gbytes of RAM. 32 
Gbytes WILL cover you for formatting major data sets, however you will 
find that when you go to production and a have a regular user base 
(20-30 people) you will need the extra 32 Gbytes for the following reasons:

1. The active user base will use MRS not only for search/lookup of 
sequences but also for BLAST/clustalw operations. These can be memory 
intensive at scale (6-7 simultaneous/queued BLAST sessions on an 8-core 
machine will require several tenths of RAM plus lookups). At that point, 
if you have only 32 Gbytes of RAM, you will need to stop your users 
until you format the datasets.

2. If you need to keep up with the biodatabanks' size increase, you will 
need the extra RAM. I compared the RSS RAM increase between EMBL 101 and 
102 operations and I saw a 4-5 % size increase in memory requirements.

In terms of disk space, to format the databases we have available here:
http://cnkeeper.uio.no:8080/mrs-web/status.do?method=databanks

we need 1.5 Tbytes in the MRS data partition, of which 1.2 Tbytes is the 
mrs dir (index). With regards to the RAID choice, yes, you certainly 
need nested RAID. Whilst 50 works, we find that 60 is a bit better in 
terms of reliability. With performance, the more stripes you have, the 
better.
