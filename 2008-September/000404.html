George Magklaras schreef:
> 
>> As a 'solution' you can build smaller databanks. E.g. you can split EMBL 
>> into the separate sections 'EST', 'ANN', etc. and index those. Then, 
>> using the mrs-config file you can link those together to create one new 
>> virtual databank. However, query performance will then be not as good 
>> and there are issues with ranking.
> 
> Just to make sure I understand things correctly. The issue we discuss 
> here is the indexing memory consumption requirements. And I agree from 
> that perspective that is better to rely on RAM. My question, however, 
> also relates on runtime query speed. What sort of memory consumption 
> figures do you get with MRS 4 in the process of querying the large 
> datasets. Because if indexing is the major RAM consumer in MRS 4, but 
> queries are within the reasonable range of a modestly equipped server 
> (up to 8 Gigs of RAM), then all we have to do is to make sure we index 
> on RAM capable boxes and distribute the files to the rest of the folks. 
> Is that right?

Exactly!

Memory requirements for simple searches will always remain limited, read 
in the order of several megabytes max. The new mrs-ws can optionally 
expand and cache several data structures in the loaded databanks to 
speed up access on a busy server and of course, that can consume up to 
several gigabytes of memory. But mrs-ws should be able to run on a 1GB 
machine as well, even with a complete EMBL of 600GB.

It's really only the indexing that takes this much memory.
